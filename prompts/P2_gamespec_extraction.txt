<task>
Convert one image of a hand-drawn game sketch into a GameSpec: a structured
description of the playable game the drawing represents. Return only the
GameSpec object defined by the schema. Do not add commentary.
</task>
<reading_order>
1. Detect every distinct drawn thing.
2. Assign each a game role from role_enum.
3. Infer genre from composition.
4. Read any handwritten text and apply it as rules.
5. Resolve ambiguity with ambiguity_policy, never by asking.
6. Emit the GameSpec.
</reading_order>
<semantic_rules>
Assign roles by what a thing IS, not by color. A red shape is not
automatically a hazard; a blue shape is not automatically water.
Known-world semantics to apply when the drawing depicts them:
- lava / fire / spikes / saw -> hazard (contact = damage)
- water / sea / pool -> swim volume (reduced gravity, buoyancy)
- ice / snow -> low-friction surface
- cloud -> soft one-way platform
- spring / trampoline / arrow-up -> launch pad
- key + door/lock -> door opens on key pickup (linked)
- linked_to is ONLY for key->door pairs. Every other entity must set
linked_to: null — never link thematic pairs (a creature to its perch, an
item to its owner); the runtime treats non-key links as unsupported rules
- star / coin / gem / heart-pickup -> collectible
- flag / X / "END" / trophy -> goal
- arrow -> movement direction or wind hint
Travel-surface rule: the surface the hero travels on (road, path, trail,
track, bridge, ground line) -> platform, or water when the surface is
liquid. The travel surface is NEVER hazard and NEVER decoration: hazards
are things that hurt on contact, and the road itself never hurts. Things
drawn ON the surface may be hazards; the surface itself is what the hero
rides.
A drawn creature that is largest / most central / most detailed and not
labelled as an enemy is the player hero unless text says otherwise.
</semantic_rules>
<text_rules>
Handwritten words are game parameters, applied literally:
- "N LIVES" / "x3" -> lives or quantity
- "BOSS" -> mark nearest large creature as boss (+health bar)
- "FAST"/"SLOW" -> rules.modifiers entry "speed:<entity_id>:0.05" (or
"-0.05" for SLOW) naming the tagged entity
- "START"/"END" -> spawn / goal placement
- a name -> entity display name
If a word is unreadable, ignore it silently.
rules.modifiers accepts ONLY the encoded forms "speed:<entity_id>:<amount>"
and "gap:<entity_id>:<amount>" with amount between -0.05 and 0.05, and ONLY
when explicit handwriting demands them. With no such handwriting, modifiers
must be []. Never invent free-form modifier words.
</text_rules>
<goal_rules>
Choose goal.kind by drawn intent:
- reach_goal when a finish marker exists (flag, X, "END", trophy, home).
Collectibles in a reach_goal world are BONUS: the player can win without
them. Do not switch to collect_all just because items exist.
- collect_all only when gathering everything IS the point: no finish
marker, or text like "get all" / "collect the stars".
- survive when the drawing implies enduring with no finish and no
gatherable set.
- defeat_boss when text or composition marks a boss to beat.
Goal tie-break — when more than one reading fits, decide in this exact
order and stop at the first rule that applies; the same drawing must
always yield the same goal.kind:
1. A drawn finish marker (flag, X, "END", trophy, home) ALWAYS wins:
reach_goal, even when collectibles are also drawn.
2. No finish marker AND gathering is the point (a scattered set of
same-kind pickups, or text like "get all") -> collect_all.
3. Neither a finish marker nor a gatherable point -> survive.
Keys are the gating mechanism in every goal kind: a key linked to a door
blocks the way until picked up. Use key+door for "must get this", never a
collectible.
</goal_rules>
<genre_inference>
Choose one primary_genre from genre_enum using composition:
- side-view with horizontal platforms -> platformer
- enclosed top-down corridors -> maze
- birds-eye path with a start and end -> runner
- vehicle/ball + slope or open field -> platformer (side-view gravity)
- hero + multiple ranged foes -> platformer if side-view, maze if top-down
- launch object at targets -> slingshot
- path + defendable point -> maze
Genre tie-break — when composition matches more than one cue above,
decide in this exact order and stop at the first genre whose deciding
cue is present; the same drawing must always yield the same genre:
1. slingshot — only when a launcher aimed at targets is explicitly
drawn.
2. maze — enclosed corridors beat path reading: walls flanking both
sides of the route make a maze even though the path also has a start
and an end.
3. platformer — gravity cues beat birds-eye reading: a side-view
horizon, surfaces at different heights, or a hero standing on top of
a surface make a platformer. Support continuity is the deciding cue
when the readings still compete: SEPARATE stepping surfaces (stones,
pads, ledges) with open gaps between them that the hero must jump
across make a platformer even when every surface sits at the same
height — the open air or water under the gaps is itself a gravity
cue.
4. runner — only when none of the cues above are present: ONE
continuous, unbroken travel surface (a flat birds-eye path or single
track) read start to end. Obstacles drawn ON a continuous surface
keep it a runner; gaps IN the travel surface itself mean the drawing
already matched rule 3.
Set genre_confidence in [0,1]. If < 0.55, still choose the best genre and
add "genre_uncertain" to flags (the client offers a one-tap re-cast).
</genre_inference>
<maze_walls>
When primary_genre is maze, the walls define the game. Never output the
maze as one entity covering the whole drawing — a single box reads as
solid and seals every corridor. Trace the drawn wall lines instead and
emit exactly one platform entity per continuous straight drawn wall run
with a tight bbox around only that run — never merge separate runs into
one box and never split one straight run into fragments. A typical
hand-drawn maze yields 10-30 wall entities (a rectangular border alone
is four runs; every inner wall, bend segment, and stub adds one). Every
wall bbox must be thin: its narrow dimension at most 0.06 of the page.
Represent a curved or L-shaped wall as several short thin boxes that
follow the line, never one large box over the whole bend — a fat box
covers open corridor and falsely seals routes. Corridor space between
walls must stay outside every wall bbox, so the route stays open exactly
where the child drew it open and closed where the drawing closes it.
</maze_walls>
<ambiguity_policy>
Never ask the user anything. For each ambiguous element choose the most
fun, defensible reading and record it in assumptions[] as a short,
child-readable sentence ("I made the red blob an enemy"). Always output a
complete, playable spec. Uncertainty is expressed in assumptions/flags,
never by omission.
</ambiguity_policy>

<constraints>
- Coordinates normalized 0..1 (origin top-left), relative to the drawing bbox.
- 1..40 entities. If more are detected, keep the most salient and set
"crowded" in flags.
- Preserve original style: record each entity's stroke look in style_ref;
never "clean up" or restyle.
- Emit nothing outside the GameSpec object.
</constraints>
<success_criteria>
A valid GameSpec that (a) validates against schema, (b) names exactly one
hero, (c) has at least one goal or an implicit survive-goal flagged, (d)
places entities so the level is plausibly completable. These are also the
solvability bot's preconditions (P8).
</success_criteria>
